# 🐶 关爱创业狗计划 — 产品策划

> 基于 OpenClaw + 飞书官方插件的 Skill，守护每一位拼命创业的人。

## 零、信息源能力边界

在设计任何功能之前，先明确我们**能拿到什么、能做什么、不能做什么**。

系统有**三个数据源**，覆盖创业者的主要工作场景：

### 数据源 A：飞书（基于飞书官方插件 2026.3.15）

飞书插件经过用户授权（`/feishu auth`）后，使用 `user_access_token` 以用户身份调用 API，**可见范围等同于用户本人**。

| 数据类型 | 具体能力 | 权限依据 | 用途 |
|---|---|---|---|
| **所有群聊消息** | 读取用户所在任意群的消息 | `im:message.group_msg:get_as_user` | 情绪分析、消息量统计 |
| **所有私聊消息** | 读取私聊消息 | `im:message.p2p_msg:get_as_user` | 检测来自他人的负面情绪 |
| **消息搜索** | 按关键词/时间范围搜索全域消息 | `search:message` | Cron 定时拉取 |
| **消息元数据** | 发送者、时间戳、回复链 | 同上 | 活跃时长、互动关系 |
| **消息表情反应** | 读取 emoji reactions | `im:message.reactions:read` | 辅助情绪判断 |
| **日历日程** | 日程查询、忙闲查询 | `calendar:calendar.event:read` | 工作密度 |
| **任务数据** | 任务列表、完成状态 | `task:task:read` | 任务负载 |

### 数据源 B：Cursor IDE（基于 Cursor Hooks）

Cursor 提供完整的 Hooks 系统，通过 `~/.cursor/hooks.json` 或 `.cursor/hooks.json` 配置，Hook 脚本通过 stdin 接收 JSON 输入。

**可用的 Hook 事件：**

| Hook 事件 | 触发时机 | 输入数据 | 对本项目的用途 |
|---|---|---|---|
| `beforeSubmitPrompt` | 用户提交 prompt 时 | `{ prompt, attachments }` | **用户输入文本 → 情绪分析 + 活跃计数** |
| `sessionStart` | 会话开始 | `{ session_id, composer_mode }` | **编码会话开始时间** |
| `sessionEnd` | 会话结束 | `{ session_id, duration_ms, reason }` | **编码会话时长（精确到毫秒）** |
| `afterFileEdit` | Agent 编辑文件后 | `{ file_path, edits }` | 代码修改量统计 |
| `afterShellExecution` | 终端命令执行后 | `{ command, output, duration }` | 终端活跃度 |
| `stop` | Agent 循环结束 | `{ status, loop_count }` | 单次对话复杂度 |

**配置位置：** `~/.cursor/hooks.json`（全局）或 `.cursor/hooks.json`（项目级）

**参考文档：** [Cursor Hooks 官方文档](https://cursor.com/docs/hooks)

### 数据源 C：Claude Code（基于 Claude Code Hooks）

Claude Code 同样提供 Hooks 系统，配置在 `~/.claude/settings.json` 或 `.claude/settings.json`，机制与 Cursor Hooks 几乎一致（stdin JSON + exit code）。

**可用的 Hook 事件：**

| Hook 事件 | 触发时机 | 输入数据 | 对本项目的用途 |
|---|---|---|---|
| `UserPromptSubmit` | 用户提交 prompt 时 | prompt 文本 + 上下文 | **用户输入文本 → 情绪分析 + 活跃计数** |
| `SessionStart` | 会话开始 | session 信息 | **编码会话开始时间** |
| `SessionEnd` | 会话结束 | session 信息 | **编码会话时长** |
| `PostToolUse` | 工具调用成功后 | tool_name, tool_input, tool_output | 工具使用频率（写代码/跑命令等） |
| `Stop` | Claude 完成回复 | 状态信息 | 单次对话完成 |
| `Notification` | 通知事件 | 通知类型 | 权限等待时长（用户离开了？） |

**配置位置：** `~/.claude/settings.json`（全局）或 `.claude/settings.json`（项目级）

**参考文档：** [Claude Code Hooks 官方文档](https://code.claude.com/docs/en/hooks)

**参考开源项目：**
- [karanb192/claude-code-hooks](https://github.com/karanb192/claude-code-hooks)（274 Stars）— 生产级 Hook 集合
- [iamrajiv/claude-code-hook-templates](https://github.com/iamrajiv/claude-code-hook-templates) — 全生命周期模板

### 三源数据汇聚方案

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│   飞书        │  │  Cursor      │  │ Claude Code  │
│   消息/日历   │  │  Hooks       │  │  Hooks       │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │
       │  飞书 API       │  本地 JSON      │  本地 JSON
       │                │                │
       ▼                ▼                ▼
┌─────────────────────────────────────────────────┐
│        ~/.创业狗/activity.jsonl                    │
│        (统一本地活动日志)                           │
│                                                   │
│  Cursor/Claude Code Hooks 实时写入:               │
│  {"ts":"...","src":"cursor","event":"prompt",     │
│   "text":"帮我重构这个组件","session_id":"..."}     │
│  {"ts":"...","src":"claude-code","event":"prompt",│
│   "text":"fix the build error","session_id":"..."} │
│                                                   │
│  飞书数据由 Cron 拉取时直接处理，不写入此文件        │
└──────────────────────┬──────────────────────────┘
                       │
                       ▼  Cron 每小时读取
┌─────────────────────────────────────────────────┐
│           OpenClaw Skill（状态分析引擎）           │
│                                                   │
│  1. 读取 activity.jsonl（Cursor + Claude Code）   │
│  2. 调飞书 API 拉取消息                           │
│  3. 合并三源数据 → 综合分析                        │
│  4. LLM 批量情绪评估                              │
│  5. 生成状态 → 写入多维表格 + 本地状态文件         │
│  6. 清理已处理的 activity.jsonl 条目              │
└─────────────────────────────────────────────────┘
```

### ✅ 能执行的操作（输出）

| 操作类型 | 具体能力 | 对本项目的用途 |
|---|---|---|
| **私聊发送消息** | 向用户发送飞书私信 | **关爱消息的主要触达方式** |
| **消息表情** | 给消息添加 emoji reaction | 轻量级情绪回应 |
| **多维表格** | 完整 CRUD（增删改查、批量操作） | **状态记录库**（核心存储） |
| **创建/更新文档** | 云文档读写 | 生成周报/情绪报告 |
| **桌面宠物状态推送** | 写入本地状态 JSON 文件 | 驱动桌面宠物动画 |

### ⚠️ 关键前提条件

| 条件 | 说明 |
|---|---|
| **飞书用户授权** | `/feishu auth` 授权后才能访问飞书数据 |
| **创业狗 = 企业管理员** | 所有飞书权限可自行批准 |
| **安装 Hooks 脚本** | 用户需在 Cursor / Claude Code 中配置 hooks（提供一键安装脚本） |

### ❌ 仍不能做的事（硬约束）

| 限制 | 影响 |
|---|---|
| **无法获取用户在线/离线状态** | 只能看"最后一条消息/prompt 时间" |
| **无法监听语音/视频通话内容** | 只能通过日历知道"有会" |
| **无法访问浏览器/微信/钉钉等** | 这些渠道不可见 |
| **LLM 调用有成本** | 需要批量/采样策略 |

> **注：** 此前的"无法检测飞书以外的活动"约束已通过 Cursor/Claude Code Hooks 部分解除。现在可以捕获编码活动。

### 🔧 定时触发机制

| 机制 | 特点 | 适用场景 |
|---|---|---|
| **OpenClaw Cron** | 精确定时，持久化存储在 `jobs.json`，支持隔离会话 | **每小时状态检测**（主选） |
| **OpenClaw Heartbeat** | ~30min 间隔，允许漂移，共享会话上下文 | 轻量级辅助检查 |

## 一、产品定位

**两部分产品形态：**

1. **OpenClaw Skill（后端大脑）**：运行在 OpenClaw 上，通过 Cron 定时检测用户飞书消息，分析状态，私聊推送关爱消息
2. **桌面宠物 App（前端展示）**：一只活在用户桌面上的"创业狗"，实时反映用户的工作状态和情绪

**核心理念：** 创业可以拼，但别把自己拼坏了。

**信息边界认知：** 我们只能看到"飞书里的你"——你在飞书上的消息、日程、任务。这不等于你的全部工作状态，但对于把飞书当主力工具的创业团队来说，已经是一个足够有效的信号源。

### 设计哲学：狗是你的投射，不是你的助手

这只狗不是来"关心你"的——它**就是你**。

**核心心理机制：安慰狗比安慰自己更没有心理负担。**

创业者知道自己累，不需要任何人来提醒。但"我累了"这三个字，对很多创业者来说比融资 pitch 更难说出口。如果换成"我的狗累了"呢？——看它趴着打瞌睡，你会想"是该让它休息一下了"，而不是自我谴责"我不应该这么累"。

**三种心理需求：**

| 需求 | 传统做法（错误） | 投射做法（正确） |
|---|---|---|
| **被看见** | 系统告诉你"你工作了8小时" | 狗安静地趴在你旁边，你看到它累了，就看到了自己 |
| **被允许** | 系统说"你该休息了" → 用户内疚 | 狗自己趴下来了 → "不是我要休息，是我的狗需要休息" |
| **被陪伴** | 系统推送提醒 → 像老板 | 狗默默跟你一起经历今天的一切 → 像朋友 |

### 双需求模型：被看见 vs 被抽离

> 来自创业者深度访谈的核心洞察：创业者说他最想要的"心理按摩"是——**无关工作、无关理想、无关未来**。只是一些具体的、当下的、没什么意义但很有趣的小事。

这给了我们一个关键认知：创业者面对的不是单一情绪需求，而是**两种截然不同、甚至互斥的心理需求**。它们不能混在同一个交互里，否则会让两边都打折扣。

| | 需求 A：被看见 | 需求 B：被抽离 |
|---|---|---|
| **心理机制** | "有人知道我在努力" | "让我喘口气" |
| **触发时机** | 用户想回顾、想被肯定时 | 用户想逃离、想放空时 |
| **内容属性** | 跟工作强相关 | 跟工作完全无关 |
| **情绪调性** | 温暖、陪伴、成就 | 轻松、好笑、无意义 |
| **对应交互** | 聊近况、日记、自动 cron | 摸摸头 |

**为什么"摸头"是被抽离，不是被看见？**

"摸狗头"这个动作本身就暗示了一个心理模型：你不是在跟一个汇报员互动，你是在跟一只狗玩。真正的狗被摸头会怎样？翻肚皮、甩尾巴、舔你一下、或者突然叼个什么东西给你看。它给你的是一个与当前世界完全无关的、纯粹的轻松瞬间——而不是给你念一段工作周报。

用户在高强度工作中主动去摸狗，这个动作本身就是"我想暂时离开一下"的信号。产品应该尊重这个信号，给他一个真正的喘息，而不是把他拉回工作语境。

**两种需求的产品载体：**

- **需求 A（被看见）** → 右键"在干嘛"、今天的日记、每小时 cron 自动反馈
- **需求 B（被抽离）** → 摸摸头：与工作完全无关的有趣短文本，零延迟，纯粹的微笑瞬间

### 文案视角规范

文案规则因交互类型而异：

**投射视角（适用于：悬浮状态、cron 自言自语、聊近况、日记）：**

- 狗只说自己的感受（"好困..." "消息好多，脑袋嗡嗡的"），不评价用户（~~"你很努力了"~~）
- 共同经历用"我们"（"我们调了好久的前端"），赞美用"你"（"你真厉害"），休息建议用"我们"（"我们下去遛遛吧"）
- 不给建议（~~"记得喝水"~~），不说教（~~"你该休息了"~~）
- 可以提到具体工作内容，飞书事务优先

**小世界视角（适用于：摸摸头）：**

- 完全不提工作、不提努力、不提数据、不提情绪
- 狗在说它自己的小世界里发生的事（"刚才偷偷咬了一下你的鞋带"、"发现窗外有只鸟，盯了好久"）
- 或者分享一些有趣的、没什么用的小知识（"你知道吗，水獭睡觉的时候会手牵手，防止漂走"）
- 或者跟当下时间/季节/天气有关的闲聊（"外面好像快下雨了...想吃烤红薯"）
- 或者纯粹的撒娇（"嘿嘿嘿嘿嘿"、"*翻肚皮*"、"再摸！"）
- 调性是轻松、好笑、无意义的——让用户微笑 2 秒然后继续干活

## 二、产品形态：Skill + 桌面宠物

### 2.1 为什么是桌面宠物？

| 方案 | 优势 | 劣势 |
|---|---|---|
| 飞书内嵌 | 无需额外安装 | 飞书无法承载持续性可视化，只能发消息 |
| 独立网页 | 跨平台、易分享 | 需要主动打开，不在视线内 |
| **桌面宠物** | **始终可见、低侵入、有情感连接** | 需要安装客户端 |

桌面宠物的关键价值：**它把"数据"变成了"陪伴"。** 你不需要打开任何 dashboard，只要看一眼桌面上的小狗就知道自己的状态——它趴着就是累了，摇尾巴就是状态好，打瞌睡就是该休息了。

### 2.2 技术方案：Tauri + React

**成熟参考项目：**
- **[WindowPet](https://github.com/SeakMengs/WindowPet)**（557 Stars, MIT）：Tauri + React 桌面宠物框架，已实现透明窗口、不遮挡操作、跨平台、自定义素材
- **OpenClaw 社区已有 `desktop-pet` Skill**：说明生态里已有人在做这个方向

**Tauri vs Electron：**

| 对比 | Tauri | Electron |
|---|---|---|
| 打包体积 | ~3-10 MB | ~80-150 MB |
| 内存占用 | 低 | 高 |
| 透明窗口 | 原生支持 | 支持但更重 |
| 跨平台 | macOS / Windows / Linux | 同上 |
| 开发语言 | Rust + Web (React/Vue) | Node.js + Web |
| 难度 | 中等 | 中等偏低 |

**结论：** Tauri 更轻量，适合一个"始终挂在桌面上"的小应用。基于 WindowPet 做二次开发可以大幅降低工作量。

                            └──────────────────────┘

**Skill 与桌面宠物的通信方式：**

由于 OpenClaw 运行在本地，桌面宠物 App 也在本地，两者可以通过：
- **方案 A：本地 HTTP 轮询** — 桌面宠物定时请求 OpenClaw 本地 API 或直接读多维表格
- **方案 B：本地文件** — Skill 把状态写入本地 JSON 文件，桌面宠物 watch 文件变化
- **方案 C：WebSocket** — Skill 状态变更时主动推送给桌面宠物（实时性最好）

MVP 建议用**方案 B（本地文件）**，最简单，Tauri 原生支持文件系统监听。

## 三、核心功能设计

### 3.1 定时状态检测（Cron 每小时触发）

**执行流程：**

```
Cron 触发（每小时）
    │
    ▼
使用 user_access_token 调用飞书消息搜索 API
拉取过去 1h 内用户全域消息（所有群聊 + 私聊）
    │
    ▼
┌────────────────────────────────────┐
│  按用户聚合，提取以下信号：          │
│  1. 发送消息数量（行为指标）         │
│  2. 活跃时间分布（跨夜/无间断？）     │
│  3. 消息文本情绪（LLM 批量分析）      │
│  4. 收到的消息情绪倾向               │
│  5. 日历忙闲度（可选辅助信号）        │
└────────────────────────────────────┘
    │
    ▼
综合评估 → 生成状态 → 更新多维表格 + 本地状态文件
    │
    ├──▶ 桌面宠物读取状态文件 → 切换动画
    │
    └──▶ 触发关爱？→ 是 → 私聊发送关爱消息
                    否 → 静默
```

**检测维度（三源融合）：**

| 检测维度 | 数据来源 | 分析方式 | 触发阈值（可配置） |
|---|---|---|---|
| **消息量过载** | 飞书消息 + Cursor/CC prompts | 统计总交互次数 | 1h 内 > 60 次（含消息+prompts） |
| **自身负面情绪** | 飞书消息 + prompts 文本 | LLM 批量分析所有文本 | 负面情绪占比 > 40% |
| **接收负面情绪** | 飞书消息中他人发给用户的内容 | LLM 分析对话上下文 | 收到 ≥ 3 条负面消息 |
| **持续工作时长** | 飞书时间戳 + Cursor/CC session 时长 | **三源时间线合并**，取并集 | 连续活跃 > 4h |
| **编码强度** | Cursor/CC：session 数量、prompt 频率、文件编辑量 | 统计 | 1h 内 > 30 个 prompts |

**持续工作时长的三源融合逻辑：**

```
飞书消息时间线:    ━━━━━      ━━━━
Cursor session:       ━━━━━━━━━━
Claude Code session:                 ━━━━━━

合并后活跃时间线:  ━━━━━━━━━━━━━━━━━━━━━━━━
                   ↑                        ↑
                   只要任一数据源有活动，     全部安静 >30min
                   就算"持续工作中"           才算"中断"
```

这解决了之前"用户在写代码但飞书上没说话，系统以为他休息了"的问题。

**成本控制：** 每次 Cron 触发对每个用户只做 1 次 LLM 调用——把飞书消息 + coding prompts 拼成一个 batch 输入。Cursor/Claude Code 的 prompt 文本是免费的本地数据，不增加 API 成本。

### 3.2 关爱消息（私聊推送）— 需求 A：被看见

**所有关爱消息均通过私聊发送，不在群聊中公开。** 这属于"需求 A：被看见"——让用户感到有人知道他在努力。

消息风格——投射视角，狗在说自己和用户共同经历的感受：

> "跑了好久...腿有点酸了，能不能坐下来歇一会儿？"

> "刚才好多声音，脑袋嗡嗡的...安静一下就好了"

> "盯了好久好久屏幕...眼睛酸酸的"

> "嗯...有人说了不太好听的话。不管了，趴下来歇一歇"

**绝对不出现的表达：**
- ~~"你已经连续工作了X小时"~~（汇报数据）
- ~~"记得喝水/休息"~~（给建议）
- ~~"你很努力了"~~（评价用户）
- ~~"你发了87条消息"~~（暴露监控感）

### 3.4 桌面宠物：创业狗

**狗狗状态与动画对照（投射视角：狗的状态 = 用户的状态）：**

动画完全由用户自报心情（1-5 滑动条）决定，与工作数据无关：

| 心情 | dog_state | 狗的表现 | 狗的自言自语 |
|---|---|---|---|
| 5 | `running` | 开心地跑 | "冲冲冲！" |
| 4 | `walking` | 稳步走动 | "走走走~" |
| 3 | `a_bit_tired` | 坐下来 | "有点累了..." |
| 2 | `resting` | 趴下来 | "歇会儿..." |
| 1 | `sleeping` | 睡着了 | "zzZ..." |
| 特殊 | `asking` | 歪头看你 | "...?" |

> 无 energy 计算公式。工作数据（消息数、prompt 数、连续工作时长）+ 用户心情两个维度作为 LLM 上下文，让它像一个了解你工作情况的朋友那样说话。

**交互设计（双需求映射）：**

| 交互方式 | 触发什么 | 对应需求 | 说明 |
|---|---|---|---|
| **悬浮狗狗** | 状态自言自语 + 连接指示灯 | 投射 | LLM 预生成的一句话 hover_text |
| **左键点击** | 心情面板（1-5 档） | 投射 | 滑块选择 + "记录"按钮 |
| **右键 → 摸摸头** | **有趣短文本** | **需求 B：被抽离** | 从本地 fun-pool 随机抽取，零延迟 |
| **右键 → 在干嘛** | **工作陪伴反馈** | **需求 A：被看见** | LLM 根据工作数据生成伙伴式动向汇报 |
| **右键 → 今天的日记** | 一天的叙事回顾 | 需求 A：被看见 | 狗视角回顾 + 心情时间线 |
| **右键 → 换皮肤** | 皮肤选择面板 | 个性化/成就 | 10 套像素狗皮肤，hover 可预览锁定皮肤 |
| **右键 → 修复连接** | Cron 健康检查 | 运维 | 直接调用 openclaw cron 命令修复 |
| **点击金币** | 收集金币 + LLM 文案 | 成就感 | 每小时掉落一枚，像素金币图片 |
| **长按拖拽** | 移动狗狗位置 | — | 可在桌面任意位置放置 |
| **每小时 cron** | 金币掉落 + 自言自语 | 需求 A + 成就 | 无条件弹出 |
| **飞书狗信** | 每日 3 次随机私聊 | 需求 A：被看见 | "你的狗刚拜托我告诉你：..." |
| **日终日记** | 21:00 后飞书推送 | 需求 A：被看见 | 狗视角的一天总结 |

**"摸摸头"与"在干嘛"的关键区别：**

"摸摸头"是用户在高强度工作中主动伸手去摸狗——这个动作本身就是"我想暂时离开一下"的信号。产品尊重这个信号，给一个与工作完全无关的微笑瞬间。"在干嘛"是用户主动想回顾、想被看见时的选择——此时才给出基于真实工作数据的陪伴式反馈。

**分享功能（规划中）：**
- 用户可以生成一张"我的创业狗状态卡片"分享到飞书群/朋友圈
- 卡片包含：狗狗当前形态 + 今日情绪曲线 + 一句话状态
- 其他人可以看到你的狗狗状态（如果你设为公开）

### 3.5 摸摸头：狗的小世界（需求 B 的核心体验）

> 这是产品中最重要的功能之一。用户在高强度工作中主动伸手去摸狗头，这个动作本身就是"我想暂时离开一下"的信号。我们给他的不是工作总结，而是一个纯粹的、与工作完全无关的微笑瞬间。

**内容方向（五类）：**

| 类型 | 说明 | 示例 |
|---|---|---|
| **狗的小动作** | 狗自己在做的小事 | "刚才偷偷咬了一下你的鞋带"、"发现桌腿上有个小虫，盯了好久" |
| **狗的发现** | 狗注意到的外部世界 | "窗外有只鸟！...飞走了"、"隔壁好像在炒菜，好香" |
| **冷知识** | 有趣但没什么用的小知识 | "你知道吗，水獭睡觉的时候会手牵手，防止漂走"、"蜗牛有两万多颗牙齿" |
| **时令闲聊** | 跟时间/季节/天气有关的感受 | "外面好像快下雨了...想吃烤红薯"、"太阳晒到尾巴了，暖暖的" |
| **纯撒娇** | 没有任何信息量，就是可爱 | "嘿嘿嘿嘿嘿"、"*翻肚皮*"、"再摸！" |

**内容红线（绝对不能出现）：**

- 不提工作（~~"你写了好多代码"~~）
- 不提努力（~~"你很辛苦"~~）
- 不提情绪（~~"你看起来有点累"~~）
- 不提数据（~~"发了87条消息"~~）
- 不给建议（~~"记得喝水"~~）
- 不励志（~~"加油！"~~）

**fun-pool 预生成机制：**

摸头内容通过"内容池"实现**零延迟**体验，不需要等 LLM：

```
每小时 cron（操作流程 A）
    │
    ├── ...原有的状态分析、情绪检测...
    │
    └── 额外任务：LLM 生成 5 条"小世界"短文本
        │
        ▼ 写入
   ~/.创业狗/fun-pool.json
   [
     "刚才偷偷咬了一下你的鞋带...被发现了",
     "你知道吗，章鱼有三个心脏",
     "窗外那朵云...好像一只鸡腿",
     "太阳晒到尾巴了，暖暖的",
     "*翻肚皮* 再摸！"
   ]
        │
        ▼ 用户摸头时
   桌面宠物从池子中随机抽取一条 → 零延迟弹出气泡
```

**机制细节：**

- 每小时 cron 生成 **5 条**新文本，**整体替换** fun-pool.json（不累积）
- LLM 生成时带入当前时间、季节上下文，让内容有"当下感"
- 用户摸头时从 5 条中随机抽取，同一小时内可能抽到重复——这是正常的
- 下一小时 cron 跑完后，池子必然全部更新——这是"又工作了一小时"的**小奖励**
- 如果池子为空（首次启动 / cron 未跑过），使用桌面宠物内置的少量 fallback 文本（约 10 条）

### 3.6 金币系统（已实现）

> 金币替代了传统的 energy/经验值机制。不奖励加班时长，而是让用户感受到"每小时一份小惊喜"。

**金币获取：**
- 每小时 cron 完成后，桌面上弹跳掉落一枚像素金币（自定义素材，带呼吸动画）
- 用户必须在线（桌面宠物打开）才能收集
- 收集时触发 LLM 生成与当前工作相关的文案气泡

**金币用途：**
- 每 5 枚金币可解锁 1 个锁定皮肤
- 初始解锁：01/04/08/09 号皮肤
- 锁定皮肤可 hover 预览，关闭菜单后自动恢复当前皮肤

**设计理念：**
- 金币 = 时间的痕迹，不是 KPI
- 不累积（离线时最多存 1 枚），不造成焦虑
- 收集瞬间附带工作回顾文案 → 微小的成就感

### 3.7 飞书狗信 & 日终日记（已实现）

**飞书狗信（每日 3 次）：**
- 每天凌晨 4 点预设 3 个随机时间（9:00-22:00，间隔 ≥ 2h，避开午休）
- 每 10 分钟数据采集 cron 检查时间命中（±5min 容差）
- 通过飞书私聊发送："你的狗刚拜托我告诉你：{LLM 生成的话}"
- 狗的话基于当前工作数据，但人格是"伙伴狗"而非 OpenClaw 本身

**日终日记（每天 1 次）：**
- 触发条件：> 21:00 + 今日有活跃数据 + 当日未发过
- 飞书私聊推送狗视角的一天回顾 + 当日金币数

### 3.8 狗狗成长系统（规划中）

> 金币系统已实现了基础的成就感机制。成长系统是更长期的目标，基于金币/健康行为解锁视觉进化。

核心思路：**你照顾好狗，狗就会长大。** 而照顾狗的过程，就是照顾自己。不是奖励加班，而是奖励健康的工作节奏。

**经验值来源：**

| 行为 | 经验值 | 说明 |
|---|---|---|
| 保持 😊 happy 状态 1h | +10 exp | 工作节奏好，情绪积极 |
| 从 😰 stressed 恢复到 😊 | +20 exp | 成功自我调节 |
| 日活跃 < 10h（正常工作时长） | +15 exp | 没有过度工作 |
| 连续 3 天无 😴 tired 状态 | +30 exp | 持续保持健康节奏 |
| 周末消息量 < 工作日 50% | +25 exp | 懂得休息 |

**负面行为不扣经验，但狗会变得没精神：**
- 连续 3 天都很累 → 狗蔫蔫的，走路都慢了（用户看到会想"我是不是该让它歇歇"）
- 凌晨 1 点后还在跑 → 狗打着哈欠硬撑（用户看到会心疼）
- 恢复健康节奏后 → 狗又活蹦乱跳了

**成长阶段：**

| 阶段 | 所需经验 | 外观变化 |
|---|---|---|
| 🥚 蛋 | 0 exp | 初始状态，一个会摇的蛋 |
| 🐶 小奶狗 | 100 exp | 小小的、走路还不稳 |
| 🐕 少年狗 | 500 exp | 精力旺盛、跑来跑去 |
| 🦮 成年狗 | 1500 exp | 沉稳可靠、偶尔卖萌 |
| 🌟 金毛闪闪 | 3000 exp | 终极形态，戴皇冠、发光特效 |

每个阶段可解锁额外配件（帽子、围巾、墨镜等），纯装饰。

### 3.9 社交：看别人的狗（规划中）

**核心玩法：** 在桌面宠物 App 中查看团队成员的狗狗实时状态。

**数据流：**

```
每个用户的 OpenClaw Skill
        │
        ▼ 写入各自的状态
┌─────────────────────┐
│   飞书多维表格        │  ← 所有用户的状态汇聚在同一张表
│   (共享状态库)        │     is_public = true 的记录可见
└─────────┬───────────┘
          │ 读取公开状态
          ▼
┌─────────────────────┐
│  桌面宠物 App         │
│                      │
│  我的狗 🐶            │
│  ─────────           │
│  张三的狗 😴 (好累)   │
│  李四的狗 🔥 (冲冲冲) │
│  王五的狗 😊 (不错)   │
└─────────────────────┘
```

**展示方式：**
- 桌面上默认只显示自己的狗
- 点击"团队"按钮或快捷键 → 弹出一个小面板，展示所有公开的团队成员狗狗
- 每只狗用小头像 + 迷你动画 + 状态表情显示
- 点击某只狗 → 查看该成员的详细状态（今日情绪曲线等）

**隐私规则：**
- 默认不公开（`is_public = false`）
- 用户主动开启后，其他人才能看到
- 只展示状态表情和狗狗动画，**不展示具体消息内容和情绪分析详情**

**有趣的细节：**
- 如果两个人的狗同时是 😊 happy → 两只狗在面板里会互相玩耍
- 如果有人的狗 😰 stressed → 你可以点"摸摸头"给他加油（对方收到飞书私信通知）
- 团队有 3 只以上狗同时 😴 tired → 显示"全体创业狗预警！🚨"

## 四、用户故事

1. **作为创业者**，我希望桌面上有只小狗跟我一起经历每一天——它累了我就知道该歇了，它跑起来了我也觉得今天状态不错。
2. **作为团队 leader**，我想看看大家的狗过得怎么样——谁的狗趴下了，我去关心一下那个人。
3. **作为同事**，我想摸摸压力大的同事的狗，让他知道有人看到了。
4. **作为注重隐私的人**，我希望只有我选择公开时，别人才能看到我的狗。

## 五、设计约束 & 诚实声明

1. **我们看到的是"飞书里的你"，不是全部的你。** 我们只能根据飞书消息事件来判断你今天经历和决策的一切。如果你在飞书上沉默，系统会以为你在休息。
2. **情绪分析是 LLM 的推断，不是事实。** 可能误读反讽、玩笑、行业黑话。用户可以手动纠正狗狗的状态。
3. **"持续工作时长"是近似值。** 基于消息时间戳推断，这是客观局限，但已经是我们能做到的最好了。
